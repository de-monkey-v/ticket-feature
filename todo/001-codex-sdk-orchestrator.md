# 001. Codex SDK Ticket Orchestrator

## 만들려는 것

이 프로젝트는 여러 로컬 프로젝트 폴더를 별칭으로 등록해 두고, 사용자가 그 별칭을 선택해서 두 가지를 할 수 있는 웹 앱을 만드는 것이 목표다.

1. `Explain` 모드에서 프로젝트 코드에 대해 질문한다.
2. `Ticket` 모드에서 기능 요청이나 수정 요청을 티켓으로 만들고, 여러 단계를 거쳐 자동으로 작업을 진행한다.

핵심 방향은 `Codex SDK`를 공통 실행 엔진으로 쓰는 것이다.  
`Explain`는 대화형 분석용, `Ticket`은 단계형 오케스트레이션용으로 사용한다.

## 사용자 경험

사용자는 프로젝트 별칭을 보고 프로젝트를 선택한다.

- Explain
  - 선택한 프로젝트 코드에 대해 질문한다.
  - 같은 프로젝트에서는 기존 thread를 이어서 대화한다.
- Ticket
  - 제목과 설명으로 티켓을 만든다.
  - 단계별 결과를 보면서 승인하거나 재실행한다.
  - 구현 이후 검증과 리뷰는 자동으로 이어진다.

## 티켓 플로우

현재 목표 플로우는 아래와 같다.

1. `Analyze`
   - 요청을 분석한다.
   - 영향 범위, 리스크, 추천 검증 항목을 정리한다.
2. `Plan`
   - 파일 단위 구현 계획을 만든다.
   - 승인 기준과 검증 계획을 만든다.
3. `Approve`
   - 사람의 승인을 기다린다.
4. `Implement`
   - 승인된 계획대로 코드를 수정한다.
5. `Verify`
   - 서버가 정해진 검증 명령을 직접 실행한다.
   - 기본 예시는 `typecheck -> test -> build`다.
6. `Review`
   - 별도 리뷰 컨텍스트에서 블로킹 이슈가 있는지 확인한다.
7. `Ready`
   - 최종 상태와 요약을 남긴다.

## 중요한 실행 원칙

- `Analyze`, `Plan`, `Review`는 구조화된 출력으로 받는다.
- `Implement`는 메인 작업 thread를 이어서 사용한다.
- `Review`는 독립성을 위해 별도 thread로 실행한다.
- 검증 명령 선택은 에이전트에게 맡기지 않고 서버 설정으로 고정한다.
- 검증 실패나 리뷰 실패가 나오면 `Implement -> Verify -> Review`를 다시 돌린다.
- 무한 루프는 막고, 구현 시도 횟수 상한을 둔다.

## 왜 이렇게 만드는가

- Explain과 Ticket이 같은 Codex 기반 동작을 공유할 수 있다.
- 계획 승인과 자동 검증을 분리해서 사람 개입 지점을 명확히 할 수 있다.
- 테스트/빌드 같은 결정적 작업은 서버가 직접 실행하는 편이 더 안정적이다.
- 리뷰를 구현 thread와 분리하면 마지막 품질 게이트 역할을 하기 쉽다.

## 현재 구현 상태

- Explain은 Codex SDK 기반으로 동작한다.
- Ticket도 Codex SDK 기반으로 동작한다.
- `Implement` 이후 `Verify -> Review -> Ready` 자동 흐름이 연결돼 있다.
- 프로젝트별 검증 명령은 `flows.config.json`에서 설정한다.
- 티켓에는 thread, 시도 횟수, 검증 결과, 리뷰 결과를 저장한다.
